概念#

关键字：唯一标识数据元素的数据项。

查找表分为静态、动态。

查找算法的评价指标：

评价一个查找算法的效率时，通常考虑查找成功/失败两种情况的ASL。

ASL的数量级反应了查找算法时间复杂度。

顺序查找#

又称==线性查找==，通常用于线性表，适用于==顺序链式、有序无序==都可。

一般的#

以顺序表、不带哨兵为例：

typedef struct {
	ElemType *elem; // 动态数组基址
	int TableLen;
}
int Search_Seq(SSTable ST, ElemType key) {
	int i;
	for (i=0; i<ST.TableLen && ST.elem[i]!=key; ++i)
	return i==ST.TableLen ? -1 : i;
}

可在0号位置设置“哨兵”，从尾部向头部挨个查找，优点是：循环时无需判断下标是否越界。

ASL：

  • 成功：$\frac{1+2+3+···+n}{n}=\frac{n+1}{2}$，O(n)
  • 失败：$n+1$，O(n)

优化的#

  1. 若==元素有序==：当前关键字大于（或小于）目标关键字时，查找失败。查找失败的ASL变小（查找成功的ASL没有变小）

    查找判定树：

    n个成功结点，n+1个失败结点。

    成功结点的查找长度 = 自身所在层数

    失败结点的查找长度 = 其父结点所在层数

  2. 若各个关键字==被查概率不等==：可按概率降序排列，使查找成功时ASL更小（查找失败的ASL没有变小）

查找效率分析：

由于查找次数不超过树高，所以时间复杂度为O(log~2~n)

插入#

在二叉排序树中插入新结点后，能够保持平衡，才能使二叉排序树的查找效率最高，即O(log~2~n)，如何保持平衡？——调整最小不平衡子树，即可使其它祖先结点都恢复平衡。

导致不平衡的四种插入情况及对应调整方式：

  • LL

  • RR

  • LR

  • RL

调整规律总结：

查找效率分析：

  • 假设深度为h的平衡二叉树最少含有n~h~个结点：n~0~=0，n~1~=1，n~2~=2，==n~h~=n~h-1~+n~h-2~+1==

    由此可得：n~3~=4，n~4~=7，n~5~=12，······

    示例：含有9个结点的平衡二叉树的最大高度为多少：4。（因为5层最少需要12个结点）

  • 由上条可证明：平衡二叉树最大深度为 O(log~2~n)，平均查找长度/时间复杂度为O(log~2~n)

删除#

平衡二叉树的删除和插入类似，都要保持二叉排序树的特性不变，且若删除结点后导致不平衡，需要调整平衡。

平衡二叉树的删除操作步骤：

  1. 删除结点（方法同“二叉排序树”）
  2. 一路向上找到最小不平衡子树，找不到就结束
  3. 找最小不平衡子树下，“个头”最高的儿子、孙子
  4. 根据孙子的位置（LL/RR/LR/RL），调整平衡
  5. 如果不平衡向上传导，回到步骤2

插入和删除#

核心要求：

  • 对m阶m树——除根结点外，结点关键字个数为 ⌈m/2⌉-1≤n≤m-1
  • 子树0<关键字1<子树1<关键字2<子树2<·······

过程：

插入过程：

  • 插入最底层终端结点，用查找来确定插入位置
  • 若插入后结点满了，则从中间位置（⌈m/2⌉）将其中的关键字分为两部分，左部分的关键字放在原结点，右部分的关键字放到新结点中，中间位置（⌈m/2⌉）的结点插入原结点的父结点中（插入位置为==指向该结点的指针右边的位置==）。
  • 若分裂后导致父结点满了，则继续分裂，直到这个过程传到根结点为止，进而导致树高+1

示例：插入93

删除过程：

  • 若删除的是非终端结点，则转化为删除终端结点，即删除其直接前驱或直接后继，找前驱后继的方法同二叉排序树的删除
  • 删除终端结点：
    • 若删除后关键字个数未低于下限，不用处理
    • 若低于下限
      • 左/右兄弟够借，则借过来顶替空缺
      • 都不够借，则与父结点内的关键字、左/右兄弟进行合并。合并后导致父结点关键字数量-1，可能需要继续合并。

散列查找#

散列表（Hash Table），又称哈希表，是一种数据结构，特点是：数据元素的==关键字==和==存储地址==直接相关

散列查找是典型的“==用空间换时间==“的算法，只要散列函数设计的合理，则散列表越长，冲突概率越低。

若不同关键字通过散列函数映射到同一个值，则为“==同义词==”

通过散列函数确定的位置已经存放了其它元素，这种情况叫“==冲突==”

装填因子α=表中记录数/散列表长度=ASL~失败~

装填因子越大，装的越满，发生冲突的可能越大，查找成功的平均查找长度也越大

常见的散列函数#

设计目标：让不同关键字的冲突尽可能少

  • 开放定址法

    • 线性探测法

      冲突后，依次探测后面的位置是否空闲。

      注意：开始求出的哈希函数值在0到12之间，而最后处理冲突后的值在0到15之间。

      查找操作：

      • 算出哈希值，依次向后比较，直到第一个空位置（该空位置的比较也算作一次比较）。
      • 比较过程中，同义词、非同义词都要被检查。
      • 越早遇到空位置，就可以越早确定查找失败。

      删除时注意（开放定址法都是这样）：不能简单地将被删元素的位置置空，否则将截断在他之后填入散列表的同义词结点的查找路径，可以做一个”删除标记“，进行逻辑删除，以便查找时，能够继续向后查找。

      由此也可以看出开放定址法的弊端：看起来很满，实际上很空

      查找效率分析：

      线性探测法冲突后再探测一定是放在某个连续的位置，所以很容易造成同义词、非同义词的”聚集（堆积）“现象，严重影响查找效率

    • 平方探测法

      能够解决线性探测法的“聚集（堆积）”问题。

      冲突后，根据平方增量序列探测位置是否空闲。

      查找操作：算出哈希值，根据平方探测序列查找目标。

      散列表长度必须是一个可以表示成4j+3的素数，才能探测到所有位置。

    • 伪随机序列法

      冲突后，根据伪随机增量序列探测位置是否空闲。
